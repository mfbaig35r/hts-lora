## Where We Started

Phase 1 left us with 149,338 formatted training examples sitting in `data/formatted/` — 119k train, 14.7k valid, 14.9k test. The data pipeline was solid: CROSS rulings extracted from our database, filtered for current-valid HTS codes, quality-checked, deduplicated, split at the ruling level (no data leakage), and formatted into chat messages with four input variants (rich, minimal, glossary-enriched, materials-only).

But two things were broken:

1. **No way to train locally.** The training code (`training/train_lora.py`) was built for CUDA — bitsandbytes 4-bit, Flash Attention 2, paged AdamW. Great for a rented A100, useless on an M4 Pro MacBook.

2. **Inference and eval code was stale v1.** The v1 system expected JSON output (`{"predicted_code": "...", "confidence": 0.95}`). But v2 training uses structured text — a hierarchy of Chapter/Heading/Subheading/HTS Code/Reasoning/Provides for. None of the evaluation machinery could parse this.

The goal for Phase 2: get a model training locally to validate the format, and rewrite inference/eval so we can actually measure results.

---

## The v2 Output Format

This is the core design decision that drives everything else. Instead of asking the model to emit JSON (brittle, easy to hallucinate malformed output), v2 uses a structured text hierarchy:

```
Chapter 18: COCOA AND COCOA PREPARATIONS
Heading 18.06: Chocolate and other food preparations containing cocoa:
Subheading 1806.90: Chocolate and preps w/cocoa, nesoi
HTS Code: 1806.90.9019

Reasoning: Classified under HTSUS 1806.90.9019 as chocolate and other food preparations...

Provides for: Chocolate and other food preparations containing cocoa, other...
```

For abstentions (when the product description is too vague):

```
Cannot classify: The product's material composition is not specified. HTS classification
often depends on the primary material (e.g., steel, plastic, textile fiber).
Please provide material details.
```

Every assistant response is wrapped in `<think>\n</think>\n\n` tags — the system prompt says "detailed thinking off" so the model skips chain-of-thought and goes straight to the hierarchy. This mirrors how Nemotron-Nano handles its thinking mode.

### parse_output.py (The Parser)

New module: `src/hts_lora/inference/parse_output.py`

A `ParsedPrediction` dataclass that captures every field the model might emit:

```python
@dataclass
class ParsedPrediction:
    chapter_code: str | None      # "85"
    chapter_desc: str | None      # "ELECTRICAL MACHINERY..."
    heading_code: str | None      # "85.44"
    heading_desc: str | None
    subheading_code: str | None   # "8544.42"
    subheading_desc: str | None
    hts_code: str | None          # "8544.42.1000"
    reasoning: str | None
    provides_for: str | None
    is_abstention: bool
    abstention_reason: str | None
    parse_ok: bool
    raw: str
```

The parser: strip think tags, check for `Cannot classify:` prefix (abstention), regex-extract each hierarchy level, handle multiline reasoning/provides_for. `parse_ok` is true if we got either an HTS code or a valid abstention. Malformed output gets `parse_ok=False` with the raw text preserved.

13 tests covering normal hierarchy, abstentions, partial output, garbage text, multiline fields.

### predict.py + batch_predict.py (Inference Rewrite)

Gutted the v1 code. Removed `TaskMode` (rerank/rag_classify/direct_classify — those are future stages), removed JSON parsing.

New `predict.py` builds messages using the same 4 input variants as training (rich, minimal, glossary_enriched, materials_only), imports `_SYSTEM_PROMPT` from formatters as single source of truth, returns `ParsedPrediction`.

`batch_predict.py` processes CSVs through the same pipeline.

### metrics.py + error_analysis.py (Eval Rewrite)

Removed: `_expected_calibration_error`, `confidence_ece`, `top_k_accuracy`, `json_parse_rate`. These were all v1 concepts (confidence scores, JSON parsing, top-k candidates).

Added: `parse_rate` (what % of outputs are parseable), `hierarchy_consistency` (does the chapter in the HTS code match the chapter the model explicitly stated?).

Error buckets still work: `CORRECT`, `WRONG_CHAPTER`, `RIGHT_CHAPTER_WRONG_HEADING`, `RIGHT_HEADING_WRONG_SUBHEADING`, `RIGHT_SUBHEADING_WRONG_SUFFIX`, `ABSTENTION`, `PARSE_FAILURE`.

### MLX Pilot Infrastructure

**configs/train_mlx.yaml** — Conservative config for 24GB M4 Pro:
- `batch_size: 1` (24GB is tight for 8B even at 4-bit)
- `lora_rank: 16`, `lora_layers: 16` (save memory vs cloud's rank 32)
- `grad_checkpoint: true` (essential)
- `mask_prompt: true` (completion-only loss, mlx-lm built-in)
- `learning_rate: 1e-5` (conservative for pilot)
- `iters: 600` (enough to see convergence on a subset)

**scripts/run_pilot_mlx.py** — Typer CLI with 4 commands:
- `convert` — handle model download (pre-quantized community model)
- `subset` — stratified sample 5k train / 500 valid from full dataset
- `train` — shell out to `python -m mlx_lm lora --train`
- `generate` — load adapter, run inference on val examples, parse output, show side-by-side table

**pyproject.toml** — new `mlx` optional dependency group.

## The Pilot Run

### Attempt 1: Model Not Found

First try used `mlx-community/Llama-3.1-Nemotron-Nano-8B-v1-4bit`. Got a 401 Unauthorized from HuggingFace. Misleading error — the repo simply doesn't exist. HF returns 401 instead of 404 for non-existent repos (fun).

Searched the HF API, found `bourn23/nvidia-llama-3.1-nemotron-nano-8b-v1-mlx-4bit`. Fixed.

### Attempt 2: CLI Deprecation

`python -m mlx_lm.lora` is deprecated in mlx-lm 0.31.x. Must use `python -m mlx_lm lora` (subcommand style). Same for convert. Fixed.

### Attempt 3: Missing --train Flag

mlx-lm 0.31.x requires explicit `--train` flag. Older versions inferred it. Error: "Must provide at least one of --train or --test". Added `--train`. Fixed.

### Attempt 4: It Works

```
Starting training..., iters: 600
Iter 1:   Val loss 2.680
Iter 100: Train loss 1.217, It/sec 0.367, Peak mem 6.251 GB
Iter 200: Train loss 0.959
Iter 300: Train loss 1.003
Iter 400: Train loss 0.928
Iter 500: Val loss 0.781
Iter 600: Val loss 0.750, Train loss 0.971
```

Val loss: 2.680 -> 0.750. Peak memory: 6.25 GB out of 24 GB available. No OOM, no thermal throttle, no issues.

### Generation Results (50 Examples)

```
Parse rate: 40/50 (80.0%)

             Sample Predictions (first 5)
+-+----------------+----------------+----------+
|#| Expected Code  | Predicted Code | Parse OK |
+-+----------------+----------------+----------+
|0| 1806909019     | 1806.90.0000   | Y        |
|1| 7610900080     | 3924.90.0000   | Y        |
|2| 9504904000     | 8501.90.0000   | Y        |
|3| 3918101000     | 4808.90.0000   | Y        |
|4| 6110202067     | 6110.30.0010   | Y        |
+-+----------------+----------------+----------+
```

**What we learned:**

1. **The format is learnable.** 80% of outputs follow the structured hierarchy correctly after just 600 iterations on 5k examples. The model learned Chapter -> Heading -> Subheading -> HTS Code -> Reasoning.

2. **Abstention works perfectly.** All 4 abstention examples in the test batch were correctly identified. The model reproduced the expected abstention text nearly verbatim — same category, same reasoning. This was the thing I was most worried about.

3. **Parse failures are repetition loops.** The 20% that failed weren't format errors — the model got stuck repeating tokens ("wallets, purses, wallets, purses, wallets...") and hit the max_tokens limit before emitting an HTS Code line. This is a known artifact of minimal training. More compute will fix it.

4. **Code accuracy is low (expected).** The model heavily favors `3924.90.0000` (plastics, nesoi) and `6110.30` (knitted apparel). It's seen ~5k examples with batch_size=1 for 600 steps — it's basically memorized the most common codes. Example 0 got the chapter right (18, cocoa) but the wrong subheading. Example 4 got the chapter right (61, knitted apparel) but the wrong heading.

5. **Descriptions are hallucinated.** The model invents chapter/heading descriptions. Chapter 48 became "SPECIAL CLASSIFICATION PROVISIONS" instead of "PAPER AND PAPERBOARD." This will improve with more training data exposure.

## The Full Dataset Run (And Why GPUs Exist)

Emboldened by the pilot, we kicked off a longer run: 3000 iterations on the full 149k dataset.

**configs/train_mlx_full.yaml** — same as pilot but `iters: 3000`, `save_every: 500`, `steps_per_eval: 500`.

Started around 2 AM. Loss curve looked great:

```
Iter 1:    Val loss 2.509
Iter 500:  Val loss 1.075, Train loss 0.831
Iter 1000: Val loss 0.767, Train loss 0.696
Iter 1300: Train loss 0.668 (last logged)
```

Then the M4 Pro hit a wall. Speed dropped from 0.37 it/sec to 0.02 it/sec — basically frozen. The process was alive (PID running, 2.4% CPU) but had accumulated only 1:25 of CPU time over 6+ hours. Thermal throttling, memory pressure, or the machine sleeping — hard to say exactly.

We killed it at iter ~1300. The iter 1000 checkpoint is saved.

**The math is clear:** At 0.37 it/sec with batch_size=1, one full epoch (149k iterations) would take ~112 hours. Three epochs = two weeks. An A100 at batch_size=8 would do three full epochs in ~2 hours.

Local MLX training validated the approach. Actual training needs a GPU.

## What's Next

1. **Rent an A100 40GB on Lambda Cloud** ($1.99/hr). Push the repo, run 3 epochs on full 149k dataset. Expected cost: ~$4-5.

2. **Expected results from proper GPU training:**
   - Parse rate: 95%+ (repetition loops disappear with more training)
   - Chapter accuracy: 70-80%+
   - Heading accuracy: 50-60%+
   - Exact 10-digit match: 20-40% (genuinely hard — even experts disagree)

3. **After first GPU run:** Evaluate on the 15k test set, analyze error patterns by chapter, tune LoRA rank / learning rate / epochs based on results.

4. **Stage 2 (future):** Contrastive boundary cases — "why this code and not that similar code." This is where the real accuracy gains live.
